# CoWater 도메인 용어 사전 (Ubiquitous Language)

**목적**: 모든 문서, 코드, 대화에서 일관된 용어를 사용하여 **도메인 이해도를 높이고 혼란을 방지합니다.**

**기본 규칙**:

- 🔧 **기술 맥락** (코드, 아키텍처, API): 영문 유지 (Mission, Task, Device 등)
- 👥 **사용자/운영 맥락** (가이드, 매뉴얼, UI): 한국어 사용 (미션, 작업, 디바이스 등)
- 📝 **혼재 필요시**: 처음 등장할 때 영문+한국어 함께 표기 가능 (관례는 정하지 않음)

---

## 1️⃣ 핵심 엔티티 (Core Entities)

### Mission (미션/임무)

| 속성        | 내용                                                                     |
| ----------- | ------------------------------------------------------------------------ |
| **정의**    | 사용자 요청을 **승인 받은 후 실제로 실행되는 완전한 작업 단위**          |
| **구성**    | 여러 개의 Task로 구성                                                    |
| **예시**    | "A 구역 고해상도 촬영 임무"                                              |
| **사용 예** | "이 임무는 10시간 소요될 예정입니다"                                     |
| **참고**    | 상태 흐름은 [schema.md#mission](core/schema.md#8-mission-실행-계획) 참조 |

---

### Task (작업/세부 단계)

| 속성          | 내용                                                                   |
| ------------- | ---------------------------------------------------------------------- |
| **정의**      | Mission을 구성하는 **개별 실행 항목**, 하나의 Device에 할당되어 실행됨 |
| **할당 대상** | Device Agent (특정 Device 제어) 또는 System Agent (시스템 처리)        |
| **예시**      | "ROV-1으로 위치 A로 이동", "고해상도 촬영 3분 실행"                    |
| **사용 예**   | "첫 번째 작업이 실패했으므로 미션이 중단되었습니다"                    |
| **참고**      | 상태 흐름은 [schema.md#task](core/schema.md#9-task-실행-단위) 참조     |

---

### Proposal (추천안/제안)

| 속성        | 내용                                                                                      |
| ----------- | ----------------------------------------------------------------------------------------- |
| **정의**    | System Agent가 제시하는 **완전한 솔루션 세트**, 사용자가 선택할 수 있는 여러 옵션 중 하나 |
| **특징**    | 여러 Task의 조합 및 순서가 **사전 검증**됨 (사용자는 조합 변경 불가)                      |
| **구성**    | 여러 개의 ProposalTask로 구성                                                             |
| **예시**    | "추천안 1: ROV로 촬영 (3시간)", "추천안 2: AUV로 촬영 (5시간)"                            |
| **사용 예** | "3개의 추천안 중 첫 번째를 선택하셨습니다"                                                |
| **참고**    | 상태 흐름은 [schema.md#proposal](core/schema.md#6-proposal-추천안) 참조                   |

---

### Device (디바이스/장비)

| 속성        | 내용                                                                |
| ----------- | ------------------------------------------------------------------- |
| **정의**    | 실제 **물리적 수중/해상 운영 기계**                                 |
| **종류**    | USV(수상무인선), AUV(자율무인잠수정), ROV(원격조종잠수정) 등        |
| **역할**    | Task를 실제로 수행                                                  |
| **능력**    | `actions[]` 필드로 수행 가능한 기능 선언 (MOVE_TO, SCAN, SAMPLE 등) |
| **예시**    | "ROV-1", "USV-Alpha"                                                |
| **사용 예** | "이 디바이스는 고해상도 촬영 기능이 없습니다"                       |

---

### Agent (에이전트)

| 속성        | 내용                                                                                       |
| ----------- | ------------------------------------------------------------------------------------------ |
| **정의**    | **Device를 제어하거나 전체를 조율하는 소프트웨어 지능체**                                  |
| **종류**    | **SYSTEM_AGENT** (전체 조율), **DEVICE_AGENT** (개별 Device 제어)                          |
| **역할**    | COMMAND, MISSION_PLANNING, ASSIGNMENT, MONITORING, POLICY_CHECK, REPORTING, DEVICE_CONTROL |
| **통신**    | 다른 Agent와 명시적으로 소통 (직접 제어 X)                                                 |
| **예시**    | "System Agent", "Device Agent for ROV-1"                                                   |
| **사용 예** | "ROV의 에이전트가 센서 오류를 감지했습니다"                                                |

---

### AgentConnection (협력 관계)

| 속성          | 내용                                                                                     |
| ------------- | ---------------------------------------------------------------------------------------- |
| **정의**      | 두 Agent 간의 **의도적 협력 관계** (통신 방식, 역할 관계)                                |
| **유형**      | RELAY(중계), COORDINATE(동기화), SHARE_DATA(데이터공유), LEADER_FOLLOWER(리더-팔로워) 등 |
| **관계 수준** | PEER (동등), PARENT_CHILD (종속)                                                         |
| **용도**      | 직접 통신 불가능한 장비를 논리적으로 연결                                                |
| **예시**      | "USV(중계) ↔ ROV(작업)" RELAY 연결                                                       |
| **사용 예**   | "수심이 깊어서 직접 통신이 불가능하므로 협력 관계를 설정했습니다"                        |

---

### Event (사건)

| 속성        | 내용                                                                 |
| ----------- | -------------------------------------------------------------------- |
| **정의**    | **시스템에서 발생한 의미 있는 변화** (Rule 실행의 트리거, 감사 기록) |
| **타입**    | USER_COMMAND, PROBLEM_DETECTED, TASK_FAILED, CRITICAL_HAZARD 등      |
| **심각도**  | INFO, WARNING, CRITICAL                                              |
| **예시**    | "LOW_BATTERY Event", "TASK_COMPLETED Event"                          |
| **사용 예** | "배터리 부족 사건이 발생하여 복귀 미션이 자동 생성되었습니다"        |

## 2️⃣ 역할 및 권한 (Roles & Permissions)

### ADMIN (관리자)

| 속성     | 내용                                                       |
| -------- | ---------------------------------------------------------- |
| **정의** | 시스템 운영 및 정책 관리 권한을 가진 역할                  |
| **권한** | Policy 생성/수정, Rule 생성/수정, Config 수정, 사용자 관리 |
| **책임** | 시스템 안정성, 정책 일관성, 규칙 검증                      |
| **예시** | "이 정책은 관리자만 추가할 수 있습니다"                    |

---

### OPERATOR (운영자)

| 속성     | 내용                                                               |
| -------- | ------------------------------------------------------------------ |
| **정의** | 일상적 임무 실행을 담당하는 역할                                   |
| **권한** | Proposal 승인, Mission 실행/취소, Device 상태 확인, Event 모니터링 |
| **책임** | 임무 성공, 안전 모니터링, 이상 상황 보고                           |
| **예시** | "운영자가 임무를 승인했습니다"                                     |

---

### VIEWER (열람자)

| 속성     | 내용                                             |
| -------- | ------------------------------------------------ |
| **정의** | 읽기 전용 접근 권한만 가진 역할                  |
| **권한** | Mission 상태 조회, Device 상태 확인, Report 조회 |
| **책임** | 없음 (모니터링만)                                |
| **예시** | "열람자는 임무 진행 상황을 볼 수만 있습니다"     |

## 3️⃣ 시스템 개념 (System Concepts)

### Planning vs Execution (계획과 실행의 분리)

| 개념          | 정의                                   | 담당                  | 결과                             |
| ------------- | -------------------------------------- | --------------------- | -------------------------------- |
| **Planning**  | 어떻게 할 것인가를 **검토/제시**       | System Agent          | Proposal 생성                    |
| **Execution** | 무엇을 **실제로 실행**                 | Device Agent + Device | Mission 진행                     |
| **의의**      | 계획 변경 없이 자동화 수준만 조절 가능 | -                     | Policy/Rule로 Level 0→3으로 진화 |

---

### Policy (정책)

| 속성               | 내용                                                                                 |
| ------------------ | ------------------------------------------------------------------------------------ |
| **정의**           | **"무엇이 허용되고, 무엇이 중요한가"를 정의하는 상위 원칙**                          |
| **예시**           | "심각한 위험은 사용자 승인 없이 자동 대응한다"                                       |
| **스코프**         | PROBLEM_DETECTION, AUTO_RESPONSE, RECOMMENDATION, APPROVAL, AGENT_CONNECTION, SYSTEM |
| **관리**           | 관리자만 생성/수정, enabled 플래그로 on/off 제어                                     |
| **여러 Rule 포함** | 하나의 Policy 아래 여러 Rule이 속할 수 있음                                          |

---

### Rule (규칙)

| 속성          | 내용                                                                                                |
| ------------- | --------------------------------------------------------------------------------------------------- |
| **정의**      | **"만약 A라면, B를 하라" 실전 판단과 트리거**                                                       |
| **구조**      | Conditions + Action                                                                                 |
| **예시**      | IF (battery < 20%) THEN (CREATE_MISSION with type=RETURN)                                           |
| **실행**      | Event 발생 시 Rule Engine이 조건 매칭 후 자동 실행                                                  |
| **관리**      | enabled 플래그, priority(실행 순서) 제어                                                            |
| **액션 타입** | CREATE_EVENT, CREATE_PROPOSAL, AUTO_CREATE_MISSION, ALLOW_AGENT_CONNECTION, REQUIRE_APPROVAL, BLOCK |

---

### Event-Triggered System (사건 기반 시스템)

| 개념       | 설명                                                                        |
| ---------- | --------------------------------------------------------------------------- |
| **원칙**   | Rule은 매 주기마다 실행 X, **의미 있는 Event 발생 시만** Rule Engine 깨어남 |
| **효율성** | 불필요한 반복 계산 제거, CPU/메모리 절감                                    |
| **추적성** | 모든 의사결정의 원인(Event)이 기록됨                                        |

## 4️⃣ 사용 규칙 및 예시

### ✅ 올바른 표기법

**기술 문서 (schema.md, ADR, principles.md 등)**:

```markdown
Mission의 상태는 READY → IN_PROGRESS → COMPLETED입니다.
Task가 FAILED되면 이후 Task는 자동으로 CANCELLED됩니다.
Device는 actions[] 필드로 수행 가능한 기능을 선언합니다.
```

**사용자/운영 문서 (매뉴얼, UI 가이드 등)**:

```markdown
이 미션은 아직 준비 상태입니다. 작업들이 준비되면 진행이 시작됩니다.
디바이스가 오류 상태이므로 현재 작업을 할당할 수 없습니다.
운영자가 추천안을 선택하면 시스템이 자동으로 임무를 생성합니다.
```

### ❌ 피해야 할 표기법

| 문제      | 예시                                            | 이유                                                    |
| --------- | ----------------------------------------------- | ------------------------------------------------------- |
| 용어 혼재 | "Task를 작업으로 번역했다가 다시 Task로 부르기" | 독자 혼란                                               |
| 역할 모호 | "사용자가 Task를 거절했다"                      | Task 거절은 Device가 (ABORTED), 사용자는 Mission을 취소 |
| 상태 오용 | "Mission이 ASSIGNED 상태"                       | ASSIGNED는 Task 상태                                    |
| 개념 혼동 | "이 Proposal을 실행하세요"                      | Proposal은 승인해야 하고, Mission을 실행함              |
